# Changelog

## 0.1.5-pre1 (2026-06-20)

### 修复

- **自身消息被自己处理（双重 Bug）**

  - **Bug 1 — MUC 消息双发**：`transport.py` 同时注册了 `"message"` 和 `"groupchat_message"` 两个 slixmpp 事件处理器。对于群聊消息两者同时触发，且 `_on_groupchat_message` 内部调用了 `self._on_message(msg)`，导致每条 MUC 消息被 `_on_payload` 发送两次。Host 端收到两条完全相同的消息，其中一条即为 Bot 自己刚发出的 echo。
    - 修复：删除 `"groupchat_message"` 事件注册及 `_on_groupchat_message` 方法，统一由 `"message"` 事件处理器根据 `msg["type"]` 区分 `chat`/`groupchat`。

  - **Bug 2 — MUC 自身消息检测失效**：`filter_pipeline.py` Step 4 将 `sender_jid`（MUC 昵称如 `"maibot"`）与 `self_id`（bare JID 如 `"bot@example.com"`）比较，类型不同永远不匹配，导致 `ignore_self_message` 对群聊完全不生效。
    - 修复：`run()` 新增 `muc_nickname` 参数。群聊时比较 MUC 昵称，私聊时保持比较 bare JID。`router.py` 从配置提取 MUC 昵称（未配置时回退到 JID 本地部分）并传入 pipeline。

- **登录后获取到已过期的消息**

  - 根因：XMPP 服务器（Prosody/Ejabberd）在 Bot 离线时存储消息，上线后通过 XEP-0203 Delayed Delivery 投递。插件完全不检查 delay 信息，所有消息不分新旧全塞给 Host。
  - 修复：注册 `xep_0203` 插件。`_on_message` 开头检测 `msg["delay"]`，带有 `<delay>` 元素的离线消息直接丢弃（仅记录 debug 日志）。

### 关于 transport.py 的调试过程（连接问题排查）

0.1.5-pre 引入了一个副作用的连接问题：Openfire 报 `XMLNotWellFormedException: Character is invalid in`，stream 层面的 XML 解析错误，导致无法建立连接。

#### 第一阶段错误的猜测：stream header 格式

- 观察：Openfire 4.7+ 使用 Netty C2S，XML 解码器较严格
- 猜测：slixmpp 默认 stream header 的 `xml:lang='en'` 属性引发解析异常
- 尝试：覆写 `_build_stream_header()`，去掉 `xml:lang` 属性
- 结果：不解决问题

#### 第二阶段错误的猜测：SSL context 兼容性

- 猜测：`ssl.create_default_context()` 在 Python >= 3.12 中默认选项与 slixmpp 的 STARTTLS 实现冲突
- 尝试：改为 `ssl.SSLContext(ssl.PROTOCOL_TLS_CLIENT)` 避免额外选项
- 结果：不解决问题

#### 第三阶段错误的猜测：connect 返回值检查

- 观察：`xmpp.connect()` 返回 `None`（异步模式表示连接进行中）
- 猜测：返回 `None` 时继续 `await xmpp.disconnected` 会永久挂起
- 尝试：加 `if not connected:` 检查，提前 `xmpp.disconnect()` + sleep
- 结果：不解决问题，且 `continue` 跳过 sleep 导致 tight loop

#### 最终原因定位：逐项 revert 法

排查：

1. 同时 revert 了 `_build_stream_header` + SSL context + connection loop → 连接恢复 ✅
2. 只有 `_build_stream_header` 被 revert，SSL context 和 connection loop 保留 → 仍故障 ❌
3. 确定问题出在 **SSL context（`SSLContext` vs `create_default_context`）** 或 **connection loop 改动** 上

最终确认：

- `ssl.SSLContext(PROTOCOL_TLS_CLIENT)` 在 `CERT_NONE` 模式下与 `ssl.create_default_context()` 行为不同：裸的 `SSLContext()` 不加载系统 CA 证书包，slixmpp 的 STARTTLS 实现可能依赖此信息做部分协商检查。缺证书时 TLS 升级静默失败，后续明文数据被当作 XML 发送给 Openfire → 非法字符报错。
- connection loop 的 `xmpp.disconnect()` 强行掐断进行中的异步连接，可能扰乱了 slixmpp 内部状态，影响下一次重连。

#### 结论

**不要改动 `transport.py` 中任何与连接建立、TLS 协商、stream header 构造相关的逻辑。** 旧版代码在这些方面经过验证可以正常工作，我们的改动应该严格限制在消息处理层（message routing、filter pipeline、codec）。

### 最终保留的改动（安全范围）

| 文件 | 改动 | 对应问题 |
|------|------|---------|
| `transport.py` | 删 `"groupchat_message"` 注册 | MUC 消息双发 |
| `transport.py` | 注册 `xep_0203` | 离线消息检测 |
| `transport.py` | `_on_message` 前段加 delay 检测 | 登录后收到过期消息 |
| `runtime/filter_pipeline.py` | `run()` 新增 `muc_nickname` 参数 | MUC 自身消息检测 |
| `runtime/filter_pipeline.py` | Step 4 改用 MUC 昵称比较 | MUC 自身消息检测 |
| `runtime/router.py` | 从配置提取 MUC 昵称并传入 pipeline | MUC 自身消息检测 |

以上改动均不涉及 TCP 连接、TLS 协商、stream header 构造等 wire protocol 层面，可以安全部署。
